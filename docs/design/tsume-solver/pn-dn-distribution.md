# pn/dn 分布の正規化プロジェクト

df-pn+ の探索品質を高めるために，WorkingTT に蓄積される pn/dn 値の分布を
対数バケット空間で正規分布に近づける取り組みのドキュメント．

---

## 1. 目的と指標

### 1.1 なぜ正規分布が望ましいか

#### DFPN における pn/dn の意味

DFPN において pn/dn は「詰みを証明/反証するために必要な探索コストの推定量」を表す．
これらは常に探索途中の **仮の値** であり，問題が未解決である限り真値には収束しない．
仮に pn/dn が早期に収束したとしたら，それはむしろ「探索アルゴリズムが問題を
効率よく解けていない」状態（同じ局面をノードバジェット内に何度も探索している）を示す．

#### WorkingTT に残るべきエントリとは

WorkingTT は探索途中の pn/dn を保持するが，理想的には以下の 2 種類のノードは
WorkingTT から **早期に追い出される** ことが望ましい:

- **詰みに近いノード (小 pn / 小 dn):** 証明または反証が完了し ProvenTT に移行する
- **詰みから遠いノード (INF 付近):** 予算超過または次の IDS ステップ開始時に除去される

WorkingTT に **長く残るべき** なのは，詰みに近いとも遠いとも断定できない
**中間的な難しさのノード**（探索が継続中であることを示す）である．

#### 正規分布が理想的な理由

上記の観点から，WorkingTT の pn/dn 分布として理想的な形は:

```
エントリ数
    ↑          ████
    │        ████████
    │      ████████████
    │    ████████████████
    └──────────────────────→ log(pn)
       0  小   中間   大  INF
         (早期   ↑    早期
         追出し 残る 追出し)
```

**中間値が多く，両端（0/INF 付近）が少ない** 正規分布型の分布．これは:
- 探索が均等に進行し，多くのノードが「まだ決着がついていない」状態にある
- 証明/反証済みのノードが WorkingTT から適切に追い出されている
- 閾値制御 (1+ε，pn_floor 等) が各ノードに適切な探索予算を割り当てている

ことを示す．

#### 値域の利用効率も重要

pn/dn の表現域は 0 から INF (u32::MAX) まであり，これを **情報量として最大限活用**
できていることも重要である．分布が狭い値域に集中している場合:

- 閾値比較の分解能が低下し，1+ε トリックの精度が落ちる
- 探索優先度の差異が表現できず，類似難易度のノードが同じ値に集まる

対数バケット空間で広い範囲に分布していること（σ が大きく，μ が全域の中央付近）は，
pn/dn が表現できる情報量を効率よく使えていることを示す．

#### まとめ: 2 つの観点

| 観点 | 理想の状態 | 現状の問題 |
|------|-----------|-----------|
| **分布の形** | 中間値が多い正規分布（両端が少ない） | 0/INF に偏重，中間値が少ない |
| **値域の利用** | 対数スケールで 0〜INF を広く使う（σ 大） | 特定バケットへの集中，σ が小さい |

### 1.2 計測方法

`scripts/analyze_pn_dn_dist.py` で 39手詰め問題を 50M ノード探索し，
WorkingTT の pn/dn 分布を IDS depth ごとに計測する．

**バケット定義:** `bucket(val) = bit_length(val)`（対数スケール，bucket 4 = 16，bucket 10 = 1024）

**正規性指標:** KL divergence（正規分布フィットとの乖離）— 小さいほど正規に近い

## 2. ベースライン計測 (v0.35.0)

### 2.1 計測結果

**計測条件:** 39手詰め，50M ノード，depth=40 IDS 終了時点

| 指標 | pn | dn |
|------|----|----|
| KL divergence (正規分布との乖離) | **1.616** | **0.695** |
| 中間エントリ (0 < val < INF) の μ | 8.6 bucket (~190, ~12S) | 5.2 bucket (~36, ~2S) |
| σ | 4.9 bucket | 0.8 bucket |
| 実質的な値域 (中間エントリ) | bucket 3〜28 (25 bucket 幅) | bucket 3〜7 (**4 bucket 幅**) |
| INF 割合 | 80.6% | 0.0% |
| 0 割合 | 0.0% | 78.5% |

**問題の整理:**

- **分布の形:** pn は bucket 5-6 に，dn は bucket 4 に集中しており，中間値が少ない
- **値域の利用:** pn は 25 bucket 幅で分布するが大半は 2 bucket 幅のスパイク．
  dn は実質 4 bucket 幅しか使っておらず，全 31 bucket の 13% しか活用できていない

### 2.2 分布の特徴

**pn 分布 (KL=1.616):**

```
エントリ数
150K ┤         ██
     │         ██
 50K ┤         ██ ██
     │    ██ ██ ██ ██ ██
 10K ┤ ██ ██ ██ ██ ██ ██ ██                           ██
     └──────────────────────────────────────────────────
       2² 2³ 2⁴ 2⁵ 2⁶ 2⁷ 2⁸ 2⁹ ...           2²⁸  bucket
```

問題点:
- **bucket 5-6 (2S〜4S) に鋭いスパイク:** `heuristic_or_pn` の離散ステップで
  `safe_escapes=1-2` のノードが 2S/4S に初期化後，探索されずに固定．
  これらのノードは「まだ決着がついていない」のではなく単に **未探索のまま** であり，
  中間値として残るべきノードではない．
- **bucket 28 付近に右テール:** WPN の AND 多段累積による極大値．
  INF には達していないが，事実上「詰みから遠い」ことを示すノードが
  INF より大幅に小さい値に留まっており，WorkingTT から追い出されにくい状態．

**dn 分布 (KL=0.695):**

```
エントリ数
260K ┤ ██
     │ ██
150K ┤ ██
     │ ██ ██
  5K ┤ ██ ██ ██ ██
     └──────────────
       2² 2³ 2⁴ 2⁵   bucket
```

問題点:
- **bucket 4 (1S=PN_UNIT) に極端な集中かつ値域が狭い:** `heuristic_dn_from_pn` の
  下限が `1S` のため，pn=64S (開放局面) が全て dn=1S に初期化される．
  σ=0.8 bucket，実質 4 bucket 幅しか使えておらず，dn の情報量が著しく低い．
  難しさが異なる局面も同一の dn=1S で表現されるため，
  AND ノードの子選択精度 (dn に基づく argmin) が低下している.

### 2.3 IDS depth による推移

| depth | 累積 nodes | total TT | pn=INF% | dn=0% | pn_μ(中間) | pn_σ(中間) |
|-------|-----------|---------|---------|------|------------|------------|
| 4 | 150K | 118 | 52.5% | 52.5% | 8.8 | 1.1 |
| 8 | 155K | 9K | 45.0% | 45.0% | 8.8 | 1.3 |
| 16 | 3.7M | 412K | 94.0% | 93.8% | 8.7 | 4.2 |
| 20 | 7.6M | 816K | 92.8% | 92.1% | 8.9 | 5.0 |
| 24 | 11.8M | 181K | 66.2% | 62.8% | 8.9 | 5.1 |
| 28 | 16.6M | 815K | 86.3% | 85.1% | 8.6 | 4.6 |
| 32 | 22.2M | 2.5M | 93.0% | 92.4% | 8.5 | 4.6 |
| 36 | 29.1M | 2.0M | 89.0% | 88.0% | 8.4 | 4.6 |
| 40 | 36.1M | 1.3M | 80.6% | 78.5% | 8.6 | 4.9 |

**観察:** depth=4〜8 で σ≈1.1〜1.3 と比較的正規に近いが，
depth=16 以降で pn=INF 割合が 90%+ まで跳ね上がり，その後改善しない．
depth=24 での一時的な改善 (66.2%) は IDS でこの深さの問題が多く解決されるため．

## 3. これまでの改善履歴

### 3.1 WPN スケールドサム (v0.29.0)

**変更:** AND pn の計算を旧式 WPN から スケールドサムへ変更

```
旧: AND pn = max(child_pn) + (unproven_count - 1) * PN_UNIT
新: AND pn = max(child_pn) + (sum(child_pn) - max(child_pn)) >> WPN_GAMMA_SHIFT
```

**効果:** 非最大子の pn 変化が親に伝播するようになり，孤立したスパイクが減少．
中間値の分布が広がり σ が拡大した (v0.29.0 以前は σ≈1〜2 と非常に狭かった)．

### 3.2 初期値の値域拡大と逆比例 dn ヒューリスティック (v0.30.0)

**変更:**
- `heuristic_or_pn` の上限を 8S → 64S に引き上げ (bucket 7 → 10)
- `heuristic_dn_from_pn` 導入: `dn = (8S)² / pn，clamp(1S, 64S)`

**効果:** pn/dn の初期値域が bucket 4〜10 に広がった．
OR dn WPN の適用が安定化し，v0.31.0 への布石となった．

### 3.3 OR ノードへの WPN 対称適用 (v0.31.0)

**変更:** OR dn にも WPN スケールドサムを適用 (AND pn と対称)

```
OR dn = max(child_dn) + (sum(child_dn) - max(child_dn)) >> WPN_GAMMA_SHIFT
```

**効果:** dn 分布の σ が拡大．dn=1S への極端な集中が緩和した (v0.30.0 の初期値拡大との相乗)．

### 3.4 CD-WPN スケールドサム (v0.35.0)

**変更:** チェーン AND ノードの pn 計算をスケールドサムへ変更

```
旧: pn = max(child_pn) + (grouped_count - 1) * PN_UNIT
新: group_rep[sq] = min(cpn to sq)
    pn = max(group_rep) + (sum(group_rep) - max(group_rep)) >> WPN_GAMMA_SHIFT
```

**効果:** チェーン AND での pn 変化の伝播精度が向上．39手詰め問題の
chain aigoma 探索で pn 分布の連続性が改善される．全 SLOW テスト pass を確認．

**影響範囲の限界:** chain AND ノードは全ノードの一部に過ぎないため，
全体 KL 値への影響は軽微であった (v0.35.0 時点: KL(pn)=1.616，KL(dn)=0.695)．

### 3.5 `heuristic_dn_from_pn` 下限引き上げ — 案 A-1 (v0.36.0)

**変更:** `heuristic_dn_from_pn` の clamp 下限を 1S → 2S に引き上げ

```rust
// Before (v0.30.0〜v0.35.0):
((C / pn.max(1) as u64) as u32).clamp(PN_UNIT, 64 * PN_UNIT)

// After (v0.36.0):
((C / pn.max(1) as u64) as u32).clamp(2 * PN_UNIT, 64 * PN_UNIT)
```

**狙い:** pn=64S (開放局面) が dn=1S (bucket 4) に固着していた問題を解消し，
dn の実質値域を bucket 4〜7 (4 幅) から bucket 5〜10 (最大 6 幅) に拡大する．

**効果 (計測待ち):** `test_no_checkmate_counter_check` 等の正常動作確認後に
`scripts/analyze_pn_dn_dist.py` で KL(dn) を計測予定．
目標: KL(dn) < 0.4，σ(dn) > 1.5 bucket 幅．

## 4. 残存課題と次のアクション

### 4.1 [課題 A] dn スパイク at bucket 4 かつ値域が 4 bucket 幅に限定

**ベースライン (v0.35.0):** dn が bucket 4 (=1S=PN_UNIT) に集中 (KL=0.695，σ=0.8，実質値域 4 bucket 幅)

**2 つの問題:**
1. **分布の形:** bucket 4 への集中で中間値が少なく，WorkingTT に残るべきノードの
   dn が他のノードと区別できない
2. **値域の利用:** dn が bucket 4〜7 の 4 幅にしか分布せず，dn の表現情報量が
   理論上の 1/8 以下しか使えていない．AND ノードの子選択 (argmin dn) の精度が低い

**根本原因:**
```rust
// heuristic_dn_from_pn: 下限が 1S のため pn=64S → dn=1S に固着
fn heuristic_dn_from_pn(pn: u32) -> u32 {
    ((C / pn as u64) as u32).clamp(PN_UNIT, 64 * PN_UNIT)
    //                              ^^^^^^ bucket 4 が下限 → 開放局面が全て dn=1S
}
```

**段階的な下限引き上げ:**

| 変更 | dn の下限 | bucket | 実質値域 (推定) | 想定 KL | 状況 |
|------|-----------|--------|----------------|---------|------|
| ベースライン | 1S (16) | 4 | 4 bucket 幅 | 0.695 | (v0.35.0 計測済み) |
| **案 A-1** | **2S (32)** | **5** | **〜6 bucket 幅** | **推定 0.5〜** | **✓ 実装済み (v0.36.0，計測待ち)** |
| 案 A-2 | 4S (64) | 6 | 〜8 bucket 幅 | 推定 0.3〜 | A-1 計測後に判断 |

**注意点:** v0.30.0 で `test_no_checkmate_counter_check` の通過を目的に
下限=1S が選ばれた経緯がある (§4.1)．引き上げ時は同テストの再確認が必須．

---

### 4.2 [課題 B] pn スパイク at bucket 5-6 かつ未探索ノードの固着

**現状:** pn が bucket 5-6 (2S〜4S) に集中 (KL=1.616)

**2 つの問題:**
1. **分布の形:** 集中ピークが中間値ではなく **未探索ノードの初期値** であるため，
   「決着がついていないから中間値に残っている」という理想とはずれている．
   初期値固着ノードは WorkingTT から早期に追い出されるか，
   探索されて値が更新されるべきである
2. **値域の利用:** bucket 5-6 に集中しており，bucket 1〜4 や bucket 7〜10 の
   初期値表現が使われていない．pn の初期情報量が低い

**根本原因:** `heuristic_or_pn` の離散ステップで `safe_escapes=1-2` のノード
(39手詰め問題の大半) が 2S/4S に固定され，探索されないまま残る．

```rust
// safe_escapes=1 → 2S, safe_escapes=2 → 4S (大半がこの2値)
| 0 | 1S | 1 | 2S | 2 | 4S | 3 | 8S | ...
```

**提案:** 中間値を追加して pn 初期値を連続化する (§3.5「S 倍率が整数に丸められている箇所」)

```
safe_escapes=1, checks≥4 → 1.5S (=24)
safe_escapes=2, checks≥4 → 3S  (=48)
safe_escapes=2, checks≤3 → 5S  (=80)
```

**注意点:** 初期 pn 変更は 1+ε 閾値を通じて探索順序全体に影響する．
必ず `test_tsume_39te_ply24_mate15_regression` 等の回帰テストで確認する．

---

### 4.3 [課題 C] pn 右テール at bucket 28

**現状:** 一部ノードが bucket 28 (≈134M) まで膨張

**根本原因候補:**
1. WPN スケールドサムの AND 多段累積 (各 AND で max + sum_other>>6 が加算)
2. 深い IDS depth での pn_floor や 1+ε による閾値伝播

**分析方針:** テール部の pn 値を持つノードが探索ツリーのどの深さ・種類のノードか
を特定するデバッグ出力を追加する (tt_diag feature 経由が適切)．

**暫定案:** `WPN_GAMMA_SHIFT` を 6 → 7 に増やすと非最大子寄与が 1/128 に減り
テールが縮まる可能性があるが，DAG 補正効果も低下する．現時点では調査優先．

---

### 4.4 [課題 D] depth=16 での pn=INF 急増

**現状:** IDS depth=16 に進むと pn=INF 割合が 45% → 94% に急増

**原因:** depth=16 では探索木が初めて深い合駒位置に到達し，解決できない局面が
大量に WorkingTT に格納される．これは探索の自然な振る舞いだが，
正規化の観点では中間値が少なくなる要因．

**対応:** 課題 A・B を先に解決し，初期値が改善されてから再計測する．
depth=16 への対処が本当に必要かどうかは再計測後に判断．

---

## 5. 対応しない事項

| 事項 | 理由 |
|------|------|
| `WPN_GAMMA_SHIFT` の削減 (6→4〜5) | 非最大子寄与増大で右テール拡大の可能性あり；DAG 補正効果との trade-off 不明 |
| `DN_FLOOR` の削減 | chain AND の pn_floor との連動あり；変更コスト・リスクが高い |
| 「正規分布を強制する」初期化 | pn が実際の探索コストを反映しなくなり探索効率が低下する |
| 並列探索 | §11.6 の方針により非採用 |

## 6. アクションプラン

```
[完了]
 └── 課題 A-1: heuristic_dn_from_pn の下限 1S→2S (v0.36.0 実装済み)
      └── 次: test_no_checkmate_counter_check 確認 → KL 計測

[優先度 高]
 ├── 課題 A 計測: analyze_pn_dn_dist.py で KL(dn) 再測定
 │    ├── KL(dn) < 0.4 なら A 完了，課題 B へ
 │    └── 改善不十分なら案 A-2 (4S) を検討
 └── 課題 B: heuristic_or_pn の中間値追加 (safe_escapes=1-2 の細分化)

[優先度 中]
 └── 課題 C: pn 右テールのノード特定 (tt_diag 追加)

[優先度 低 / 再計測後判断]
 └── 課題 D: depth=16 の pn=INF 急増への追加対策
```

各ステップ後に `scripts/analyze_pn_dn_dist.py` で KL divergence を計測し，
**KL(pn) < 1.0，KL(dn) < 0.4** を近期目標とする．
最終的な有効性は 39手詰め backward 解析 (§10.2) のノード数改善で評価する．
