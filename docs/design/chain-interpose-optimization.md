# チェーン合駒最適化

## 概要

チェーン合駒(連続合い駒)は，飛び駒(飛車・角・香)による遠距離王手に対して，
玉と飛び駒の間のマスに駒を打つ(合駒)防御手のうち，飛び駒がその合駒を
取り進むことで再び王手になり，さらに合駒が可能になる再帰的構造を指す．

チェーン合駒は詰将棋ソルバーにおける主要なボトルネックの一つである．
n マスのチェーンに対して各マスで k 種の合駒が可能な場合，
最悪 O(k^n) の分岐が発生する．

### 動機: 39手詰め問題

以下の39手詰め問題がこの最適化の主な対象である:

```
SFEN: 9/1+R+N1kP2S/6pn1/9/9/5+B3/1R2S4/3p5/9 b NPb4g2sn4l14p 1
```

ply 24 で銀(5g)が6fへ移動し，飛車(8g)の横利きを開く
開き王手(discovered check)が発生する．
飛車(8g)と玉(1g)の間に6マス(7g, 6g, 5g, 4g, 3g, 2g)の合駒スペースがあり，
うち5マス(7g, 6g, 5g, 4g, 3g)がチェーンマスに分類される．
この構造が探索の指数的爆発を引き起こし，1Mノード/3分の予算では解けない．

## 背景: 合駒マスの分類

飛び駒による王手に対する合駒マス(between squares)は，
以下の3カテゴリに分類される:

### 通常マス (Normal)

守備側(玉以外)の利きが存在する，または合駒マスが玉に隣接しており
飛び駒が取り進んだ後に玉の逃げ道がある場合．
通常マスへの合駒は全駒種(歩→香→桂→銀→金→角→飛)を生成する．

### 無駄合いマス (Futile)

以下のすべてを満たすマス:
- 守備側(玉以外)の利きがない
- 合駒マスが玉に隣接していないか，隣接していても飛び駒が取り進んだ後に玉の逃げ道がない
- 玉側から見てブレークポイント(通常マス)より手前にある

無駄合いマスへの駒打ちは完全にスキップされる．

### チェーンマス (Chain)

futile の条件を満たすが，ブレークポイントより飛び駒側(遠い側)にあるマス．
チェーンマスへの合駒は，飛び駒が取り進んだ後に再び王手となり，
さらなる合駒が可能になる再帰的な構造を生む．
チェーンマスへの駒打ちは3カテゴリの代表駒に限定される(§3.2参照)．

## 実装済み最適化

以下は `rust/maou_shogi/src/dfpn.rs` に実装済みのチェーン合駒最適化である．

### 3.1 Futile/Chain 分類

**実装箇所:** `compute_futile_and_chain_squares` (dfpn.rs:3960-4024)

玉と飛び駒の間の各マス(between squares)を走査し，futile と chain に分類する．

**判定ロジック:**

1. 各マスについて以下を判定:
   - 守備側(玉除く)の利きがあれば → 通常マス(取り返せる)
   - 合駒マスが玉に隣接し，攻め方の他駒の利きがなければ → 通常マス(玉が取り返せる)
   - 合駒マスが玉に隣接し，飛び駒が取り進んだ後に逃げ道があれば → 通常マス
   - いずれにも該当しなければ → futile

2. 玉側からチェッカー方向へ走査し，最後の通常マス(ブレークポイント)より
   チェッカー側の futile マスを chain に再分類

**補助関数:** `king_can_escape_after_slider_capture` (dfpn.rs:4026-4080)
飛び駒が合駒マスに取り進んだ状態をシミュレートし，玉の逃げ道を判定する．

### 3.2 チェーンドロップ3カテゴリ制限

**実装箇所:** `generate_chain_drops` (dfpn.rs:4253-4309)

チェーンマスへの駒打ちを以下の3カテゴリから各1手に制限する:

| カテゴリ | 駒種 | 代表の選択 |
|---------|------|----------|
| 前方利き系 | 歩→香→銀→金→飛 | 最弱の合法駒1つ |
| 斜め利き系 | 角 | 角のみ |
| 跳躍系 | 桂 | 桂のみ |

**根拠:** 前方利き系では，弱い駒で詰みが証明できれば強い駒でも証明できる
(攻め方が合駒を取った後，手に入る駒が強いほど詰ませやすい)．
この包含関係により，最弱の駒1つで代表できる．
角と桂は利きの方向が異なるため独立のカテゴリとなる．

**効果:** 合駒マスあたりの駒打ち数を最大7手から最大3手に削減．

**呼び出し元:** `generate_interpositions` (dfpn.rs:4109-4245) が
チェーンマスへの合駒生成時にこの関数を呼び出す(dfpn.rs:4216)．

### 3.3 合駒遅延展開 (Deferred Children)

**実装箇所:** `mid()` 内の子ノード初期化 (dfpn.rs:2196-2332)

AND ノードの合駒(駒打ち)を即座に展開せず，`deferred_children` に分離する．

**仕組み:**

1. AND ノードの子を分類 (dfpn.rs:2196-2226):
   - 駒移動(玉逃げ・紐付き合駒) → `children`(即座に展開)
   - 駒打ち(合駒) → `deferred_children`(遅延)

2. 非合駒の応手を先に探索し，TT に証明を蓄積

3. `children` が空になったら `deferred_children` から1手ずつ活性化 (dfpn.rs:2330-2332):
   弱い駒から順に活性化し，証明済みのTTエントリを強い駒の探索で援用する

**効果:** 非合駒応手で反証できれば合駒の展開自体を回避．
合駒展開時も逐次活性化により不要な分岐を抑制．

### 3.4 合駒事前証明 (Interpose Pre-Solve)

**実装箇所:** `interpose_pre_solve` (dfpn.rs:3157-3288)

**設定:** `interpose_pre_solve_nodes: u64` (デフォルト: 256ノード/合駒，dfpn.rs:1231)

**仕組み:**

1. メインTTを一時TTにスワップ (dfpn.rs:3170-3173)
2. 各合駒を小予算(256ノード)で `mid()` 実行 (dfpn.rs:3212-3219)
3. 証明済みの合駒は `deferred_children` から除去
4. 証明済みエントリのみをメインTTにマージ (dfpn.rs:3258-3264)

**判定結果:** (dfpn.rs:3283-3287)
- `AllProved`: 全合駒が証明済み → AND ノードを即座に証明
- `Disproved`: いずれかの合駒が反証 → AND ノード全体が反証
- `Partial`: 一部未証明 → `deferred_children` に残す

**適用範囲:**
通常マスへの合駒に対してのみ実行される．チェーンマスのみで構成される
AND ノードでは Pre-Solve をスキップし，PNS/IDS の自然な探索に委ねる
(§3.10, §3.11 参照)．これはチェーン合駒に対する Pre-Solve が
一時TTのオーバーヘッドと中間結果の破棄により逆効果となるためである
(計算量分析の節を参照)．

**効果:** 通常マスへの合駒に対して，短時間で証明可能な合駒を除去．
残りの合駒のみが本探索で展開される．

### 3.5 TT ベース合駒プレフィルタ

**実装箇所:** `try_prefilter_block` (dfpn.rs:3301-3384)

**呼び出し:** 合駒を `deferred_children` に追加する前 (dfpn.rs:2204-2207)

**仕組み:**

1. 合駒を盤上で実行
2. 攻め方の全合法手から，合駒マスへの捕獲かつ王手になる手を探索
3. 捕獲後の局面をメインTTで参照 (dfpn.rs:3340)
4. pn=0(証明済み)なら合駒の OR ノードも証明 → 展開不要

**IDS との相乗効果:**
浅い IDS 反復でチェーン末端の証明がTTに蓄積され，
深い反復では浅いレベルの合駒がプレフィルタで即座にスキップされる．
これによりチェーン合駒がボトムアップに折り畳まれる．

### 3.6 同一マス証明転用

**実装箇所:** `cross_deduce_same_square_proofs` (dfpn.rs:3401-3505)，
`cross_deduce_deferred` (dfpn.rs:3577-3700)

**呼び出し:**
- `interpose_pre_solve` 内で合駒 i が証明された後 (dfpn.rs:3237-3240)
- メイン探索の MID ループ内で合駒が証明された後 (dfpn.rs:2807-2813)

**仕組み:**

同一マス S への合駒 P1, P2, ..., Pn は，攻め方が捕獲した後の盤面
(position_key)が全て同一になる(異なるのは攻め方の持ち駒のみ)．

1. 合駒 i が証明済みになった後，同一マスの未解決合駒 j を列挙
2. 合駒 j の捕獲後の攻め方持ち駒を計算 (dfpn.rs:3470-3472):
   `hand_j = base_hand - solved_piece + piece_j`
3. TT で捕獲後局面を参照: `look_up(pc_pk, &hand_j, pc_remaining)`
4. pn=0 なら合駒 j も証明 → `deferred_children` から除去

**効果:** 同一マスの合駒間でTTエントリを相互利用し，重複探索を削減．

### 3.7 DFPN-E エッジコスト

**実装箇所:** `edge_cost_and` (dfpn.rs:107-119)

AND ノードの応手に以下のコストを付与:

| 応手の種類 | コスト | 理由 |
|-----------|--------|------|
| 駒打ち(合駒) | 0 | 攻め方が取り進むだけで有利 |
| 玉逃げ・駒移動 | 1 | 中立 |
| 駒取り | 2 | 攻め方の駒が減り不利 |

**効果:** コスト加算により合駒の初期 pn が小さくなり，
探索優先度が下がる(OR ノードでは pn が大きい子を後回しにする)．

### 3.8 Hand Dominance による TT 再利用

**実装箇所:** `hand_gte_forward_chain` (dfpn.rs:412-426)，
`TranspositionTable::look_up` (dfpn.rs:605-648)

**仕組み:**

TT 参照時に持ち駒の完全一致ではなく包含関係(dominance)で判定する:

- **証明済み(pn=0)**: 攻め方の持ち駒が TT エントリ以上なら再利用可能
  (持ち駒が多いほど詰ませやすい)
- **反証済み(dn=0)**: 攻め方の持ち駒が TT エントリ以下なら再利用可能
  (持ち駒が少ないほど詰ませにくい)

**前方チェーン補填 (forward chain):**
歩 ≤ 香 ≤ 飛 のカスケード補填を実装(dfpn.rs:421-425):
不足する歩を香で，不足する香を飛で代替可能とみなす．
桂・銀・金・角は独立判定(代替関係なし)．

**効果:** 同一盤面で持ち駒が異なるエントリを広く再利用し，
チェーン合駒の捕獲後局面のTTヒット率を向上させる．

### 3.9 合駒 DN バイアス

**実装箇所:** 定数 `INTERPOSE_DN_BIAS = INF / 2` (dfpn.rs:47)，
適用箇所 (dfpn.rs:2727)

AND ノードの合駒(駒打ち)の初期 dn に `INF/2` を加算し，
探索優先度を下げる．実質的には deferred_children による遅延展開(§3.3)が
主要な制御手段であり，DN バイアスは補助的な役割．

### 3.10 チェーンマス内→外の探索順序

**実装箇所:** `mid()` 内のチェーンドロップ順序付け (dfpn.rs:2287-2301)

チェーンマスの合駒を `deferred_children` に追加した後，
玉に近い側(内側，残りチェーンが短い)から飛び駒に近い側(外側，
残りチェーンが長い)の順にソートする．

**仕組み:**

全ての `deferred_children` がチェーンマスへの合駒である場合，
各合駒のチェビシェフ距離(玉からの最大座標差)でソートする (dfpn.rs:2295-2300):

```
deferred_children.sort_by_key(|(m, _, _, _)| {
    let to = m.to_sq();
    let dr = (to.row() as i8 - king_sq.row() as i8).unsigned_abs();
    let dc = (to.col() as i8 - king_sq.col() as i8).unsigned_abs();
    dr.max(dc) // Chebyshev distance: inner (small) first
});
```

チェーンマス S1(内側), S2, ..., Sn(外側) への同一駒種の合駒について:
- S1 の合駒を先に証明すると，捕獲後の局面(飛車が S1 に移動し
  S2...Sn がチェーンとして残る)のTTエントリが蓄積される
- S2 の合駒の捕獲後は S3...Sn がチェーンとして残り，
  S1 証明時のTTエントリの部分集合を再利用できる

**効果:** 短いサブチェーンから順に証明が蓄積され，
長いサブチェーンの探索時にTT再利用効率が向上する．

### 3.11 チェーン合駒の PNS/IDS 統合 (Pre-Solve スキップ)

**実装箇所:** `mid()` 内のチェーン判定 (dfpn.rs:2286-2302)

チェーンマスのみで構成される AND ノードでは `interpose_pre_solve` を
スキップし，PNS と IDS の既存の仕組みでチェーン合駒を処理する．

**背景:**

チェーン合駒に対する Pre-Solve は以下の問題を持つ(計算量分析の節参照):

- 合駒の数だけ `mid()` が走る(5チェーンマス × 3カテゴリ = 15回)
- 一時TTは空から開始されるため，メインTTのエントリを活用できない
- 未証明エントリは破棄され，探索努力が無駄になる
- IDS 反復ごとに繰り返されるが，前回の中間結果は引き継がれない

**代替手段:**

チェーン合駒は以下の既存最適化で十分にカバーされる:

1. **`try_prefilter_block` (§3.5)**: IDS の前回反復でメインTTに蓄積された
   証明を活用し，合駒を `deferred_children` に追加する前にフィルタ
2. **`deferred_children` の逐次活性化 (§3.3)**: 弱い駒から順に1手ずつ
   活性化し，PNS の閾値制御で探索量を自然に調整
3. **`cross_deduce_deferred` (§3.6)**: 合駒が証明された際に
   同一マスの他の合駒をメインTT上で転用
4. **IDS のボトムアップ特性**: 浅い IDS 反復で末端の短いチェーンが証明され，
   深い反復で浅いレベルの合駒がプレフィルタでスキップされる

通常マスへの合駒に対しては従来の Pre-Solve を維持する(チェーン構造を
持たないため256ノードで証明できる可能性が高い)．

**効果:** IDS 反復ごとの Pre-Solve オーバーヘッド排除により NPS が改善される．
特に深いチェーン構造(5マス以上)で効果が顕著．

## 計算量分析: Pre-Solve の指数的オーバーヘッド

この節では，チェーン合駒に対して Pre-Solve を適用した場合の
計算量を分析する．この分析が §3.11(チェーン合駒の Pre-Solve スキップ)
の根拠となっている．

### 問題の構造

n マスのチェーンに対して各マスで k 種の合駒が可能な場合，
Pre-Solve を含むメイン探索の計算量は最悪 O(k^n) となる．
以下にその導出を示す．

### AND ノード数の漸化式

チェーンマス S1(内側), S2, ..., Sn(外側) について，
マス Si への合駒に対して飛び駒が取り進むと，
残りのチェーンは S1, ..., S(i-1) の i-1 マスとなる．

AND ノードでは全ての合駒を証明する必要があるため，
チェーン長 m に対する探索量 W(m) は:

```
W(0) = 1  (チェーンなし，ベースケース)
W(m) = k × Σ_{j=0}^{m-1} W(j)
```

ここで k は各マスの合駒の種類数(3カテゴリ制限により最大3)．
Σ は各マスへの合駒がそれぞれ異なる長さのサブチェーンを生む
ことに対応する:

- 最外側(Sm)への合駒 → 残りチェーン m-1 マス → W(m-1)
- 次の外側(S(m-1))への合駒 → 残りチェーン m-2 マス → W(m-2)
- ...
- 最内側(S1)への合駒 → 残りチェーン 0 マス → W(0)

### 閉形式の導出

S(m) = Σ_{j=0}^{m} W(j) とおくと:

```
W(m) = k × S(m-1)
S(m) = S(m-1) + W(m) = S(m-1) + k × S(m-1) = (1+k) × S(m-1)
```

S(0) = W(0) = 1 より:

```
S(m) = (1+k)^m
W(m) = k × (1+k)^{m-1}
```

### Pre-Solve の総コスト

各 AND ノードで Pre-Solve が実行され，最大 k×m 回の `mid()` を
各256ノード予算で呼び出す．`in_pre_solve` フラグ(dfpn.rs:2288)は
Pre-Solve 内の `mid()` がネストした Pre-Solve を呼ぶことを防ぐが，
メイン探索での再帰的な Pre-Solve 呼び出しは防げない:

1. レベル1 AND ノード: Pre-Solve が k×n 回 `mid()` を実行
2. 256ノード予算では深いチェーンの証明に不十分 → `deferred_children` に残留
3. メインMIDループが残った合駒を探索 → 飛び駒が取り進む
4. レベル2 AND ノード到達: `in_pre_solve = false` → 再び Pre-Solve 実行
5. 以降レベル n まで再帰

Pre-Solve 内の `mid()` 総呼び出し回数:

```
Σ_{各ANDノード} (そのノードの合駒数) × 256ノード
= O(k × n × W(n))
= O(k² × n × (1+k)^{n-1})
```

### IDS による増幅

IDS の各反復で同じ AND ノードに再到達するたびに Pre-Solve が
再実行される．一時 TT は毎回空から開始されるため(dfpn.rs:3170-3173)，
前回の Pre-Solve で証明に至らなかった中間結果は引き継がれない．

IDS の反復回数を D とすると，Pre-Solve の総コストは:

```
O(D × k² × n × (1+k)^{n-1})
```

### 具体例: 39手詰め問題

ply 24 のチェーン構造(n=5, k=3):

```
W(5) = 3 × (1+3)^4 = 3 × 256 = 768 AND ノード
```

各 AND ノードで最大 15 回(5マス × 3カテゴリ)の Pre-Solve `mid()` が
実行されるため，1回の IDS 反復だけでも:

```
768 × 15 × 256 ≈ 2,949,120 ノード相当の Pre-Solve オーバーヘッド
```

これが NPS の極端な低下(通常 ~数千 NPS → 250~960 NPS)の直接的原因であり，
チェーン合駒に対する Pre-Solve スキップ(§3.11)の根拠となっている．

## 未実装の最適化案

### チェーン深さ DN スケーリング

**概要:**
チェーンマスへのドロップの初期 DN をチェーン内の位置に応じて
スケーリングし，内側(浅いサブツリー)を先に探索させる．

**仕組み:**
現在の `INTERPOSE_DN_BIAS` は全ドロップ合駒に一律 `INF/2` を適用する．
チェーンマスの場合，外側(飛び駒に近い)マスほど取り進み後の
サブツリーが深く，探索コストが高い．

提案: チェーン内の位置 d (0=内側, n-1=外側) に応じて
`INTERPOSE_DN_BIAS * (d + 1)` を初期 DN に加算する．

§3.10(内→外の探索順序)と同じ方向性だが，`deferred_children` の順序変更ではなく
DN 値による動的な優先度制御で実現する点が異なる．
§3.10 と組み合わせることで相乗効果が期待できる．

**正確性への影響:** なし．探索順序のみの変更．

**期待効果:** 中

**実装難度:** 低

探索効率は改善されているが，深いチェーンにおけるNPS低下は
依然として残存する課題である．
