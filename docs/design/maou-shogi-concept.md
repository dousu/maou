# maou_shogi 設計思想

## 概要

`maou_shogi`は，cshogiの機能をRustで再実装したクレートである．
単なるcshogiの置き換えではなく，cshogiが扱えない問題領域をカバーしつつ，
呼び出し側のリソース設定次第であらゆる将棋の問題を現実的な速度で解けることを目指している．

## コンセプト

### 1. cshogiが解けない問題をカバーする

cshogiの詰将棋ソルバーには以下の制約がある:

- **片玉局面に非対応**: 玉が1つしかない局面(詰将棋の標準形式)を扱えない
- **中合い(合駒)の考慮不足**: 合駒による防御手順を正しく探索できない
- **最短経路の未考慮**: 複数の詰み手順がある場合に最短手順を保証しない

maou_shogiはこれらすべてに対応する:

- 片玉局面を正しく処理
- 中合いを含む全ての合法的な防御手段を探索
- `find_shortest`オプションによる最短手順の探索

### 2. どんな詰将棋問題も現実的な速度で解ける

呼び出し側がリソースパラメータ(`depth`, `nodes`, `timeout`)を増やすことで，
実質的にどんな詰将棋問題にも対応できる設計とする．

```python
# 短手数の問題: デフォルト設定で高速に解ける
solve_tsume(sfen, depth=31, nodes=1_048_576)

# 長手数の問題: リソースを増やして対応
solve_tsume(sfen, depth=200, nodes=100_000_000, timeout_secs=600)
```

制限に達した場合は`Unknown`を返し，呼び出し側がリソースを増やして再試行できる．
ソルバー自体に本質的な手数制限や問題制限はない．

### 3. cshogiとのパフォーマンス関係

#### 詰将棋ソルバー

中合い等の考慮によりcshogiよりノード数が増加するのは設計上許容する．
ただし，cshogiが取り入れていない高度な枝刈り手法を積極的に導入し，
場合によってはノード数が少なく高速で解けることを目指す．

**現在実装済みの枝刈り・最適化:**

| 手法 | 説明 |
|------|------|
| 持ち駒の優越(hand dominance) | 転置表で持ち駒の包含関係を利用した証明/反証の再利用 |
| 反復深化 df-pn | 浅い探索の証明エントリで深い探索を加速(depth > 15で有効) |
| インライン詰み検出 | 1手・3手の詰みをDf-Pnオーバーヘッドなしで検出 |
| 合駒後回し(INTERPOSE_DN_BIAS) | AND nodeで非合駒応手を先に展開しTTを温める |
| 無駄合い・連続合い枝刈り | 紐のない合駒をスキップ，連続合いは代表駒種のみ生成 |
| 単一子最適化 | 合法手が1手の場合にMIDループをバイパス |
| df-pn+式閾値戦略 | εを加算してスラッシング(閾値振動)を抑制 |
| 証明駒・反証駒 | 持ち駒の必要十分条件を追跡し，TTヒット率を向上 |

**今後導入を検討する枝刈り:**

| 手法 | 期待効果 |
|------|---------|
| ORノード探索優先度の改善 | 王手手の順序付けによる早期証明 |
| 反復深化の深さ刻み・予算配分の改善 | 中間イテレーションの効率向上 |
| 優等局面の活用強化 | 証明/反証の伝播を拡大 |

#### 詰将棋以外

合法手生成，SFEN処理，HCPエンコード，特徴量抽出など，
詰将棋以外の全ての機能ではcshogi相当あるいはcshogi以上のパフォーマンスを目指す．
Rustによる実装と最適化(Bitboard演算，インクリメンタルZobristハッシュ等)により，
多くの操作でcshogiを上回る性能を実現している．

## アーキテクチャ上の位置づけ

```
maou (Python)
└── maou._rust.maou_shogi (PyO3バインディング)
    └── maou_shogi (Rustクレート)
        ├── Board / Position   ← 局面表現
        ├── movegen            ← 合法手生成
        ├── dfpn               ← 詰将棋ソルバー
        ├── feature            ← NN特徴量
        ├── hcp                ← HCPコーデック
        └── sfen               ← SFEN処理
```

- cshogiとのデータ互換性(駒ID，Move16/32エンコーディング，HCPフォーマット)を維持
- Python側はmaou_shogi経由で全ての将棋操作にアクセス
- 将来的にcshogi依存を完全に除去可能
